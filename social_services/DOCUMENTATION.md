## 1. Общая информация о системе

### 1.1 Назначение

Система предназначена для автоматизации учёта социальных услуг в стационарном учреждении социального обслуживания (дом-интернат для престарелых и инвалидов). Система рассчитана на работу с учреждением вместимостью до 500 проживающих.

### 1.2 Основные задачи

- Ведение базы данных получателей социальных услуг (ПСУ)
- Учёт оказанных социальных услуг в табельной форме
- Формирование индивидуальных планов предоставления социальных услуг (ИППСУ)
- Генерация актов оказанных услуг для бухгалтерии
- Отслеживание перемещений проживающих между отделениями
- Формирование отчётов и печатных форм

### 1.3 Технологический стек

| Компонент | Технология |
|-----------|------------|
| Backend | Django 5.x |
| База данных | PostgreSQL (продакшен) / SQLite (разработка) |
| Frontend | Tailwind CSS, HTMX |
| Генерация PDF | WeasyPrint |
| Статические файлы | Whitenoise |
| Контейнеризация | Docker, Docker Compose |

## 2. Архитектура проекта

### 2.1 Структура приложений

Проект построен по модульной архитектуре с разделением на 4 приложения:

```
social_services/
├── apps/
│   ├── core/           # Ядро системы
│   ├── recipients/     # Проживающие и ИППСУ
│   ├── services/       # Услуги и табель
│   └── reports/        # Генерация отчётов
├── config/             # Настройки Django
├── templates/          # Шаблоны HTML
├── static/             # Статические файлы
└── media/              # Загруженные файлы
```

### 2.2 Описание модулей

#### apps.core
- Пользователи системы с ролевой моделью
- Отделения учреждения
- Middleware для контроля доступа
- Главная страница дашборда

#### apps.recipients
- Получатели социальных услуг (ПСУ)
- История статусов и перемещений
- Договоры (ИППСУ)
- Месячные данные по доходам

#### apps.services
- Справочник социальных услуг
- Категории услуг
- Периодичность оказания услуг
- Табель учёта услуг
- Расписание услуг по отделениям
- Блокировка табелей

#### apps.reports
- Генератор актов оказанных услуг
- Печатные формы актов

## 3. Модели данных

### 3.1 Приложение core

#### Department (Отделение)
| Поле | Тип | Описание |
|------|-----|----------|
| name | CharField | Название отделения |
| code | CharField | Уникальный код |
| department_type | CharField | Тип: residential, mercy, hospital, vacation, deceased |
| is_mercy | BooleanField | Признак отделения милосердия |
| capacity | PositiveIntegerField | Количество мест |
| description | TextField | Описание |

**Типы отделений:**
- `residential` — обычное отделение проживания (1, 2, 3, 4 и т.д.)
- `mercy` — отделение милосердия
- `hospital` — больница (для временного размещения)
- `vacation` — отпуск (виртуальное отделение)
- `deceased` — умер (виртуальное отделение для выбывших)

#### User (Пользователь)
| Поле | Тип | Описание |
|------|-----|----------|
| username | CharField | Логин |
| first_name | CharField | Имя |
| last_name | CharField | Фамилия |
| patronymic | CharField | Отчество |
| role | CharField | Роль: admin, hr, specialist, medic |
| department | ForeignKey | Привязка к отделению |

## 4. Система ролей и прав доступа

### 4.1 Роли пользователей

| Роль | Код | Описание прав |
|------|-----|---------------|
| Администратор | admin | Полный доступ ко всем данным и функциям |
| Отдел кадров | hr | Просмотр всех данных, редактирование справочников |
| Специалист | specialist | Доступ только к своему отделению |
| Медик | medic | Доступ только к своему отделению |

### 4.2 Матрица доступа

| Функция | admin | hr | specialist | medic |
|---------|-------|-----|------------|-------|
| Просмотр всех проживающих | ✓ | ✓ | ✗ | ✗ |
| Просмотр своего отделения | ✓ | ✓ | ✓ | ✓ |
| Редактирование справочника услуг | ✓ | ✓ | ✗ | ✗ |
| Заполнение табеля | ✓ | ✓ | ✓ | ✓ |
| Редактирование ИППСУ | ✓ | ✓ | ✗ | ✗ |
| Изменение статуса проживающего | ✓ | ✓ | ✗ | ✗ |
| Блокировка табеля | ✓ | ✓ | ✗ | ✗ |

### 4.3 Middleware доступа

[`DepartmentAccessMiddleware`](social_services/apps/core/middleware.py:9) контролирует доступ пользователей к данным:
- Пропускает неавторизованных пользователей на страницы авторизации
- Пропускает админов и кадровиков без ограничений
- Ограничивает медиков и специалистов только своим отделением

## 5. Функциональные модули

### 5.1 Модуль проживающих (recipients)

#### Список проживающих
- Фильтрация по отделению
- Фильтрация по типу отделения (статусу)
- Поиск по ФИО (регистронезависимый для кириллицы)
- Сортировка по полям: фамилия, имя, дата рождения, отделение, комната
- Пагинация (50 записей на страницу)

#### Карточка проживающего
- Редактирование личных данных
- Загрузка фотографии
- Изменение отделения/комнаты
- История статусов
- История перемещений
- Список договоров (ИППСУ)
- Статистика оказанных услуг

#### Управление ИППСУ
- Создание/редактирование договора
- Выбор услуг из справочника
- Группировка услуг по категориям
- Активация/деактивация договора

#### Изменение статуса
- Перевод между отделениями
- Перевод в больницу/отпуск
- Отметка о выбытии (смерть)
- Автоматическое создание истории

### 5.2 Модуль услуг (services)

#### Справочник услуг
- Иерархическая структура (услуги и подуслуги)
- Группировка по категориям
- Настройка периодичности
- Установка цены
- Лимиты на количество в месяц

#### Табель учёта услуг
**Основные возможности:**
- Выбор года и месяца
- Выбор отделения и проживающего
- Отображение только услуг из ИППСУ
- Ввод количества услуг по дням
- Автосохранение через HTMX
- Подсветка выходных и праздников
- Отображение итогов по услугам
- Контроль лимитов

**Автозаполнение табеля:**
- По расписанию услуг отделения
- Для ежедневных услуг
- С учётом лимитов на месяц
- Проверка статуса проживающего

**Блокировка табеля:**
- Защита от случайного редактирования
- Блокировка по месяцам
- Разблокировка администратором

#### Печать табеля
- Полная печатная форма
- Все дни месяца
- Итоги по услугам
- Группировка по категориям

### 5.3 Модуль отчётов (reports)

#### Генератор актов
- Выбор проживающего
- Выбор периода (месяц, год)
- Ввод данных о доходе и пенсии
- Автоматический расчёт:
  - 75% от дохода (лимит)
  - Итоговая сумма услуг
  - Разница между пенсией и услугами

#### Печатная форма акта
- Список оказанных услуг
- Количество и цена
- Итоговая сумма
- Данные о доходах
- Сквозная нумерация строк
- Группировка по категориям

## 6. API интерфейсы

### 6.1 API услуг

#### Сохранение записи табеля
```
POST /services/api/service-log/
```
**Параметры:**
- `recipient_id` — ID проживающего
- `service_id` — ID услуги
- `day` — день месяца
- `month` — месяц (1-12)
- `year` — год
- `quantity` — количество

**Ответ:**
```json
{
    "success": true,
    "quantity": "2",
    "total": "15",
    "max_quantity": 30
}
```

#### Batch-режим (заполнение строки)
```
POST /services/api/service-log/
```
**Дополнительные параметры:**
- `batch` = "true"
- `days` — JSON-массив дней

#### Очистка месяца
```
POST /services/api/clear-month/
```
**Параметры:**
- `recipient_id`
- `year`
- `month`

#### Очистка дня
```
POST /services/api/clear-day/
```
**Параметры:**
- `recipient_id`
- `year`
- `month`
- `day`

#### Автозаполнение
```
POST /services/api/autofill/
```
**Параметры:**
- `recipient_id`
- `year`
- `month`

**Ответ:**
```json
{
    "success": true,
    "filled_count": 150,
    "message": "Заполнено 150 записей"
}
```

#### Блокировка табеля
```
POST /services/api/toggle-lock/
```
**Параметры:**
- `recipient_id`
- `year`
- `month`

### 6.2 API проживающих

#### Список по отделению
```
GET /recipients/api/by-department/<department_id>/
```
**Ответ:**
```json
{
    "recipients": [
        {
            "id": 1,
            "last_name": "Иванов",
            "first_name": "Иван",
            "patronymic": "Иванович",
            "room": "101"
        }
    ]
}
```

#### Изменение статуса
```
POST /recipients/<pk>/change-status/
```
**Параметры:**
- `department_id` — ID нового отделения
- `room` — номер комнаты
- `reason` — причина
- `admission_date` — дата заселения
- `discharge_date` — дата выбытия

## 9. Технические особенности

### 9.1 HTMX интеграция

Табель использует HTMX для:
- Автосохранения без перезагрузки
- Динамического обновления итогов
- Отображения ошибок валидации

### 9.2 Обработка кириллицы

Особенности работы с кириллицей:
- Регистронезависимый поиск в Python
- Регистронезависимая сортировка в Python
- Корректное отображение в PDF

### 9.3 История цен

При сохранении записи табеля цена услуги копируется в поле `price_at_service`:
- Сохраняется история изменения цен
- Корректный расчёт сумм за прошлые периоды

### 9.4 Пагинация

- 50 записей на страницу
- Применяется в списке проживающих

### 9.5 Медиа-файлы

- Фотографии проживающих: `media/recipients/photos/{id}/`
- Автоматическое создание директорий
- Поддержка форматов: JPG, PNG, GIF

## 10. Развертывание и эксплуатация

### 10.1 Требования

**Системные требования:**
- Python 3.10+
- PostgreSQL 13+ (для продакшена)
- Docker и Docker Compose (для контейнеризации)

**Python-зависимости:**
```
Django>=5.0
django-htmx
whitenoise
psycopg2-binary
WeasyPrint
Pillow
```

### 10.2 Переменные окружения

| Переменная | Описание | По умолчанию |
|------------|----------|--------------|
| SECRET_KEY | Секретный ключ Django | — |
| DEBUG | Режим отладки | True |
| ALLOWED_HOSTS | Разрешённые хосты | * |
| DB_NAME | Имя базы данных | social_services |
| DB_USER | Пользователь БД | postgres |
| DB_PASSWORD | Пароль БД | postgres |
| DB_HOST | Хост БД | localhost |
| DB_PORT | Порт БД | 5432 |

### 10.3 Команды управления

#### Создание демо-данных
```bash
python manage.py setup_demo
```
Создаёт:
- 6 пользователей (admin, hr, medic1-4, mercy)
- 5 отделений
- 38 социальных услуг
- 500 проживающих с ИППСУ

#### Миграция данных
```bash
python migrate_sqlite_to_pg.py
```
Перенос данных из SQLite в PostgreSQL.

### 10.4 Резервное копирование

**База данных:**
```bash
# PostgreSQL
pg_dump social_services > backup.sql

# SQLite
cp db.sqlite3 db_backup.sqlite3
```

**Медиа-файлы:**
```bash
tar -czvf media_backup.tar.gz media/
```

### 10.5 Мониторинг

**Логирование:**
- Логи Django в stdout
- Ошибки API логируются с трассировкой

**Производительность:**
- Использование `select_related` для оптимизации запросов
- Bulk-операции для массового обновления
- Кэширование статических файлов через Whitenoise
