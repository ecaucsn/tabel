# Алгоритм работы функции автозаполнения табеля

## Общая информация

**Функция:** `autofill_tabel`  
**Файл:** `apps/services/views.py`  
**Метод:** POST  
**URL:** `/services/api/autofill/`

## Назначение

Автоматическое заполнение табеля учёта услуг для указанного получателя за указанный месяц на основе:
1. Услуг из ИППСУ (Индивидуальный план предоставления социальных услуг)
2. Назначенных услуг из модуля "Назначение услуг"
3. Расписания отделения

## Источники данных

### 1. ИППСУ (ContractService)

```python
contract_services = ContractService.objects.filter(
    contract__recipient=recipient,
    contract__is_active=True
).select_related('service')
```

**Описание:** Услуги, закреплённые за получателем в активных договорах.

### 2. Назначенные услуги (ServiceRecipient)

```python
assigned_services = ServiceRecipient.objects.filter(
    recipient=recipient,
    is_active=True
).select_related('service')
```

**Описание:** Услуги, назначенные получателю через модуль "Назначение услуг" с указанием периодичности.

### 3. Расписание отделения (ServiceSchedule)

```python
schedules = ServiceSchedule.objects.filter(
    department=recipient.department
).select_related('service')
```

**Описание:** Расписание оказания услуг для отделения по дням недели.

### 4. Существующие записи (ServiceLog)

```python
existing_logs = ServiceLog.objects.filter(
    recipient=recipient,
    date__year=year,
    date__month=month
)
```

**Описание:** Уже проставленные записи услуг за указанный месяц.

## Обработка периодичности

### Для услуг из ИППСУ

| Условие | Алгоритм |
|---------|----------|
| Есть расписание отделения | Заполнение по дням недели из расписания |
| Нет расписания, периодичность "ежедневно" | Заполнение каждый день с количеством `times_per_period` |
| Нет расписания, другая периодичность | Пропуск услуги |

### Для назначенных услуг

| Периодичность | Алгоритм заполнения |
|---------------|---------------------|
| `daily` | Каждый день месяца, количество = 1 |
| `weekly` | Первый понедельник месяца, количество = 1 |
| `biweekly` | 1-е и 15-е число месяца, количество = 1 |
| `monthly` | 1-е число месяца, количество = 1 |
| `quarterly` | 1-е число, если месяц ∈ {3, 6, 9, 12}, количество = 1 |
| `custom` | 1-е число месяца, количество = 1 |

## Лимиты и ограничения

### max_quantity_per_month

Если у услуги указан `max_quantity_per_month`, автозаполнение не превысит это значение:

```python
max_quantity = service.max_quantity_per_month
if max_quantity:
    if service_totals[service_id] >= max_quantity:
        continue  # Пропустить запись
```

### Приоритет источников

1. **ИППСУ** — обрабатывается первым
2. **Назначения** — обрабатывается вторым, пропускает услуги, уже обработанные через ИППСУ

## Обработка существующих записей

### Обновление vs Создание

```python
key = (service_id, current_date)
if key in existing_logs_dict:
    # Обновить существующую запись
    log = existing_logs_dict[key]
    log.quantity = quantity
    log.provider = user
    log.price_at_service = service.price
    logs_to_update.append(log)
else:
    # Создать новую запись
    logs_to_create.append(ServiceLog(
        recipient=recipient,
        service=service,
        date=current_date,
        quantity=quantity,
        provider=user,
        price_at_service=service.price
    ))
```

## Сохранение данных

```python
with transaction.atomic():
    if logs_to_create:
        ServiceLog.objects.bulk_create(logs_to_create, ignore_conflicts=True)
    if logs_to_update:
        ServiceLog.objects.bulk_update(
            logs_to_update, 
            ['quantity', 'provider', 'price_at_service']
        )
```

**Особенности:**
- Используется транзакция для атомарности
- `ignore_conflicts=True` — игнорирует дубликаты при создании
- Массовое обновление для оптимизации производительности

### Успешное выполнение


### Ошибки

| Код | Причина | Сообщение |
|-----|---------|-----------|
| 403 | Нет прав доступа | "Нет доступа" |
| 403 | Табель заблокирован | "Табель заблокирован. Редактирование невозможно." |
| 200 | Статус не active | "Невозможно автозаполнение: статус проживающего '...'" |

## Изменения в версии 2.1

Добавлен учёт истории статусов проживающего:

1. Загрузка истории изменений из `RecipientHistory`
2. Определение неактивных дней (отпуск, больница, выбыл)
3. Пропуск заполнения услуг за неактивные дни
4. Поддержка частичных периодов (например, отпуск с 10 по 20 число)

### Вспомогательные функции

#### `get_inactive_days_in_month(recipient, year, month)`

Возвращает множество дней месяца, в которые проживающий был неактивен.

**Логика работы:**
1. Загружаются все записи истории до конца целевого месяца
2. Определяется статус на начало месяца (последняя запись до 1-го числа)
3. Проход по всем дням месяца с отслеживанием изменений статуса
4. Дни с неактивным статусом добавляются в результат

**Неактивные статусы:**
- `vacation` — отпуск
- `hospital` — больница
- `discharged` — выбыл

### Пример работы

Если проживающий был в отпуске с 10 по 20 февраля:
- Дни 1-9: услуги проставляются
- Дни 10-20: услуги НЕ проставляются
- Дни 21-28: услуги проставляются
